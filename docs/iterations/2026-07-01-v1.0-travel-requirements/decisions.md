# v1.0 旅行需求决策

## D1：H5 和原生编辑器并存

决定：H5 只负责公共模板的精美浏览，原生小程序负责实例编辑和协作。

原因：保留现有 HTML 视觉成果，同时避免把权限、云存储、活动记录和地图编辑塞入 H5。

## D2：单一模板数据源

决定：从 HTML 提取版本化结构数据，H5 与小程序种子均由它生成。

原因：当前 HTML `trip` 比 `tokyoTravelTemplate.js` 丰富，继续双份维护会持续漂移。

## D3：原生详情承载主操作

决定：`使用此计划` 放在原生模板详情页，H5 不是创建实例的唯一入口。

原因：`web-view` 加载或通信失败时，核心转化链路仍然可用。

## D4：候选地点池进入 P0

决定：旅行 v1.0 支持待安排地点。

原因：它是成熟旅行规划产品中连接“收藏”和“每日排期”的关键步骤，且能与后续菜谱待安排池复用概念。

## D5：路线优化留在 P1

决定：P0 只保证排序、地图展示、交通方式和打开地图。

原因：自动距离、耗时和最优顺序依赖地图服务、配额和异常处理，不应阻塞基础编辑闭环。

## D6：菜谱不进入旅行 v1.0 spec

决定：菜谱调研继续保留，但本轮只生产旅行 spec。

原因：菜谱与旅行可以共享计划模型，但用户流程、数据结构和验证方式不同，同时实现会放大 v1.0 范围。

## D7：版本快照不作为 spec 输入

决定：`docs/versions/v1.0/` 用于版本阅读和范围快照，后续 spec 只读取本聚合迭代。

原因：避免 PRD、版本文档和迭代文档同时演化导致冲突。

## D8：公开模板不包含敏感数据

决定：确认号、证件、成员、内部预算等只存在于空间实例。

原因：H5 是公共只读模板，不具备空间权限边界。
